# Сервисы и Инструменты

Раздел **Services** содержит описание ключевых инструментов фреймворка, которые разработчик использует для реализации бизнес-логики бота. Если **Engine** — это "скелет" системы, то **Services** — это её "мышцы" и "органы чувств".

---

## 🛠 Обзор сервисов

| Сервис | Назначение | Ключевая особенность |
| :--- | :--- | :--- |
| **[Director](./director/README.md)** | Навигация | Автоматически переключает сцены и FSM-статусы. |
| **[ViewSender](./view_sender/README.md)** | Отображение | Синхронизирует сообщения, создавая эффект SPA. |
| **[FSM & States](./fsm/README.md)** | Память | Изолирует данные фич друг от друга (Namespacing). |
| **[Redis Streams](./redis/README.md)** | События | Позволяет боту реагировать на внешние асинхронные сигналы. |
| **[Base Orchestrator](./base/README.md)** | Логика | Стандартный интерфейс для написания кода фич. |
| **[Animation](./animation/README.md)** | UI-эффекты | Визуализация процесса ожидания ответа. |
| **[Helpers & Utils](./helper/README.md)** | Утилиты | Быстрое получение ID, работа с контекстом и подпись URL. |

---

## 🧩 Взаимодействие сервисов

Сервисы в **codex-bot** спроектированы так, чтобы работать в тесной связке, оставаясь при этом независимыми (Stateless).

**Стандартный флоу:**
1. **Director** получает контекст от мидлвари.
2. **Director** запрашивает нужную фичу у **Container**.
3. **Orchestrator** выполняет логику, используя **FSM** для данных.
4. Результат передается в **ViewSender** для отрисовки в Telegram.
