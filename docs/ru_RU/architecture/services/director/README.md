# Director — Координатор и Мозг навигации

**Director** — это центральный архитектурный компонент **codex-bot**. Если Оркестраторы — это рабочие инструменты, то Директор — это прораб, который знает, куда отправить пользователя, какие данные ему передать и как подготовить для этого почву.

Он создается динамически для каждого входящего апдейта и сопровождает запрос на всем пути: от мидлвари до отправки сообщения.

---

## 🧠 Smart Resolver (Умная навигация)

Это механизм, превращающий вашего бота в **Backend-driven** систему. Вместо того чтобы прописывать логику переходов (`if response == "banned": ...`) в каждом хендлере, вы делегируете это Директору.

### Концепция Конверта (Envelope)
Директор ожидает данные в формате «Конверта», где метаданные навигации отделены от бизнес-данных:
```python
{
    "meta": {
        "__next_scene__": "banned",  # Инструкция для Директора
        "reason": "spam"
    },
    "payload": {
        "user_id": 123,              # Полезные данные для оркестратора
        "name": "Ivan"
    }
}
```

### Применение в коде
Метод `director.resolve(data)` анализирует этот конверт. Если в `meta` найден ключ редиректа, Директор:
1.  Останавливает текущий поток логики.
2.  Выполняет переход (`set_scene`) на новую фичу.
3.  Передает ей очищенный `payload`.

Это позволяет бэкенду управлять интерфейсом бота «на лету», просто добавляя заголовки или ключи в JSON-ответ.

---

## 🚀 Глубокая логика set_scene

Метод `set_scene` — это не просто вызов другой функции. Это сложный процесс синхронизации:

1.  **Поиск Оркестратора**: Директор обращается к DI-контейнеру и достает нужный инстанс фичи по ключу.
2.  **Идемпотентное управление FSM**:
    *   Директор проверяет `orchestrator.expected_state`.
    *   Если стейт пользователя в Redis отличается от требуемого, Директор меняет его.
    *   Если стейт уже верный — Директор **не делает лишних запросов** к базе, экономя ресурсы.
3.  **Изоляция Контекста**: Директор передает в оркестратор `self` (самого себя). Оркестратор получает доступ к `user_id`, `chat_id` и `container`, но не может мутировать глобальное состояние приложения.
4.  **Автоматическое Обогащение DTO**: На выходе из оркестратора часто получается «голый» `ViewResultDTO` (текст и кнопки). Директор ловит его и превращает в `UnifiedViewDTO`, автоматически проставляя:
    *   `chat_id` и `session_key` (чтобы ViewSender знал, куда слать).
    *   `trigger_message_id` (для автоматического удаления команды `/start` или кнопок).

---

## 🛠 Почему это Stateless?

Директор **не хранит** состояние внутри себя между запросами. Он живет ровно столько, сколько обрабатывается одно нажатие кнопки. Это позволяет вам запускать 100 копий бота одновременно: любому инстансу Директора достаточно данных из входящего `Update`, чтобы восстановить полную картину мира и принять решение.

## 🧭 Пример сложного сценария

```python
async def handle_click(callback: CallbackQuery, director: Director):
    # 1. Получаем данные (Smart Resolve)
    raw_response = await director.container.api.post_action(director.user_id)
    result = await director.resolve(raw_response)

    # 2. Если бэкенд велел перейти (например, на оплату) — resolve вернет UnifiedViewDTO
    if isinstance(result, UnifiedViewDTO):
        await director.container.view_sender.send(result)
        return

    # 3. Иначе продолжаем логику текущей фичи
    view = await director.render(payload=result)
    await director.container.view_sender.send(view)
```
